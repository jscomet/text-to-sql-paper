# SQL-MARS: 面向用户模糊需求的 Text2SQL 结构化数据推荐系统

徐嘉雯 1 郑云贵 1 周 伟 1 徐尧强 2 胡卉芪 1 周 烜 1

1华东师范大学数据科学与工程学院上海 200062

2国家电网有限公司华东分部 上海 200120

(51275903078@stu.ecnu.edu.cn)

摘 要 随着大语言模型 (Large Language Model, LLM) 技术的成熟,基于自然语言的数据库交互系统(如 Chat2DB、ChatExcel)已实现广泛应用。然而,现有系统普遍依赖于"精确查询"假设,难以应对现实场景中广泛存在的模糊需求,即用户需要在与系统交互的过程中明确其查询需求。为了应对这个挑战,本文提出 SQL-MARS (SQL-oriented Multi-Agent Recommender System),该系统基于"感知-行动-评估"闭环机制的多智能体协同框架,实现面向数据库模糊查询需求的动态判别与自适应处理。系统提出三层元数据架构建模用户需求以实现模糊感知。在此基础上系统实现数据导航功能,根据用户的模糊需求分粒度向用户推荐查询建议,渐进式引导用户澄清查询需求。同时,系统提出外部资料与本地数据融合机制,充分利用外部资料中有价值的信息。此外,我们还创建了Bird-fuzzy 模糊需求数据集,系统实现了自动化评估。实验结果表明,SQL-MARS 能够识别模糊需求并有效引导用户澄清数据需求。

关键词: 模糊需求: 多智能体协作: 元数据分层: 数据导航: 外部资料融合

# SQL-MARS: Text-to-SQL Structured Data Recommendation System for Ambiguous User Requirements

XU Jiawen<sup>1</sup>, ZHENG Yungui<sup>1</sup>, ZHOU Wei<sup>1</sup>, XU Yaoqiang<sup>2</sup>, HU Huiqi<sup>1</sup> and ZHOU Xuan<sup>1</sup>

1 School of Data Science and Engineering, East China Normal University, Shanghai 200062, China

2 East China Branch of State Grid Corporation of China, Shanghai 200120, China

Abstract With the maturity of LLM technology, natural language-based database interaction systems (e.g., Chat2DB, ChatExcel) have achieved wide application. However, existing systems generally rely on the "precise query" assumption and struggle to handle the ubiquitous ambiguous requirements in real-world scenarios, where users need to clarify their query needs during interaction with the system. To address this challenge, this paper proposes SQL-MARS (SQL-oriented Multi-Agent Recommender System), a multi-agent collaborative framework based on a "perception-action-evaluation" closed-loop mechanism for dynamic identification and adaptive processing of ambiguous database query requirements. The system introduces a three-layer metadata architecture to model user's requirements for ambiguous awareness. Based on this, it implements data navigation function, providing query recommendations at varying granularities based on users' ambiguous requirements to progressively guide them in clarifying their query needs. Additionally, the system proposes the fusion mechanism between external knowledge and local data to fully utilize valuable information from external sources. It also creates the dataset named Bird-fuzzy for ambiguous requirements and implements automated evaluation. Experimental results show that SQL-MARS can effectively identify ambiguous requirements and guide users to clarify their data needs.

Keywords Ambiguous requirements, Multi-agent collaboration, Hierarchical metadata, Data navigation, External knowledge fusion

# 1 引言

随着大语言模型(LLM)技术的成熟,数据库智能 交互得到了支持,Chat2DB[1]等系统已实现自然语言查 询功能部署。然而,这些系统普遍建立在"精确查询" 的理想化假设基础上,而现实应用场景中的用户查询 往往包含模糊语义。例如,用户在一开始并不明确想 要查询的数据具体如何表达;又例如,用户在表达"高 分"时量化阈值缺失或未明确定义"就业率"对象范围等。 在应对此类模糊查询时,由于结构化数据库存储范式 与 LLM 对需求的语义解析之间存在本质性差异,无法 实现精准映射,导致系统既无法准确识别查询需求中 的模糊性,又缺乏有效的交互机制引导用户澄清需求。 此外,在查询过程中,现有系统处理用户上传的外部 资料时也未能有效将其与本地数据库融合,难以充分 利用资料中蕴含的有价值信息来辅助查询。

举例而言,在医疗健康领域,模糊查询的普遍性 严重制约了基于自然语言交互的诊疗决策系统的准确 性和效率。以临床医生提出"找出需要重点关注的患者" 这一典型需求为例,系统需要处理的模糊性至少存在 两个维度:在对象范畴上,重症监护场景中可能指"生 命体征不稳定的患者",而在门诊随访中可能特指"复 查结果异常的患者";在指标定义层面,对于上述的生 命体征指标,心内科可能关注心率变异性(HRV), 呼吸科则可能侧重血氧饱和度。这些模糊的维度若不 加以澄清会直接影响到最终查询结果与需求的一致性。 同时,对于医生上传的相关资料,无法将其与当前数 据库系统关联起来(比如资料中包含正常血氧饱和度 等指标,而本地呼吸科数据库中也存在该指标)。为 了推测医生模糊查询的意图,用户不得不通过多轮的 试错式对话逐步明确需求(确认查询对象、异常指标、 阈值定义等),这种交互模式不仅大幅增加时间成本, 而且对非数据库专业的用户造成严重认知负荷。究其 本质,现有的交互式数据查询系统存在如下缺陷:

# **1**)缺乏需求模糊度判别机制;**2**)缺乏引导用户 渐进式澄清的框架;**3**)对于与本地数据库相关的外部 资料利用率低下。

为改进这些缺陷,我们需依次实现:语义建模用 户需求,对齐数据库结构化数据;基于语义对齐结果 感知需求模糊度;基于数据导航框架渐进式引导用户 澄清数据需求。在澄清过程中,需要跨领域数据库引 导,我们发现实体多义性导致的 SQL 生成幻觉现象亟 需解决。简而言之,本文将面向用户模糊需求的数据 库结构化数据导航总结为以下三项任务:

- **1**)语义对齐建模:建立用户自然语言表达与数据 库结构化数据的映射体系,消除语义鸿沟;
- **2**)模糊感知与渐进式澄清:创建需求模糊度感知 机制,根据需求模糊度通过动态反馈多粒度查询意见 的方式与用户交互,引导其渐进式澄清需求;
- **3**)**SQL** 生成抗幻觉优化:增强元数据语义,构建 领域约束体系,解决实体多义性导致的 SQL 生成问题。

本文提出了 SQL-MARS 系统(SQL-oriented Multi-Agent Recommender System),其技术路径围绕数据库 模糊查询的全生命周期处理展开:

首先,针对任务 1),本系统提出了对元数据进行 分层操作以对用户需求进行建模的方法。通过构建"库-表-字段"三级元数据网络,建立层级化语义映射体系, 实现自然语言查询需求与数据库模式的端到端的语义 对齐。

其次,针对任务 2),本系统提出了基于多智能体 协 同 的 数 据 导 航 框 架 。 该 框 架 基 于 四 大 智 能 体 (Planner, Advisor, Seeker 和 Evaluator)实现了"感知-行 动-评估"的闭环处理流程:感知过程,Planner 通过多 粒度元数据探针实现需求模糊度的动态检测;行动过 程采用条件触发机制,根据模糊感知结果自适应选择 工作流,需求模糊时 Advisor 基于分层元数据实施多粒 度需求推荐策略以引导用户澄清需求,而需求明确时 Seeker 负责 SQL 生成并执行查询检索;评估过程, Evaluator 综合客观指标(交互轮数、响应延迟、上下 文合理性反思评估)和主观指标(用户满意度)实现 系统性能的量化评估。

此外,本文在 Advisor 智能体内部设计了基于文档 解析器的异构数据融合工具,实现外部非结构化文档 与本地数据库元数据的语义对齐,旨在从外部资料中 提取有价值的指标,为 Advisor 查询建议生成补充知识 源,提高推荐质量。

然后,针对任务 3),本系统提出了基于元数据语 义增强的方法解决 SQL 生成过程中的幻觉现象。采用 细粒度驱动的"语义反哺"策略,由字段信息增强表语 义,再由表语义反向增强数据库语义,生成包含领域 标记、场景标签和语义约束信息的增强元数据。在 SQL 生成过程中抑制因跨领域数据库实体多义性导致 的语义歧义问题。

最后,在对系统评估阶段,我们发现面向模糊需 求的评估基准存在明显断层,Spider、Bird[2]等主流数 据集仅针对明确需求设计。因此,需要构建一个面向 模糊需求的评估数据集。我们创建了 Bird-fuzzy 模糊需 求数据集,用于系统自动化评估。

本文的主要研究贡献可归纳为以下五个方面:

- **1**)基于分层元数据的查询需求澄清方法:构建 "库-表-字段"三级语义映射体系,实现自然语言与数据 库模式的端到端对齐。
- **2**)基于多智能体协同的数据导航框架:设计一种 基于"感知-行动-评估"闭环机制的多智能体协同框架, 实现面向数据库模糊查询需求的动态判别与自适应处 理。
- **3**)融合外部资料与本地数据的数据探索问题推荐 机制:设计基于文档解析器的异构数据融合方案,实 现外部非结构化文档与本地数据库元数据的语义对齐, 提取有价值的指标以辅助 Advisor 推荐查询建议。
- **4**)基于元数据语义增强的**SQL**生成抗幻觉方法: 提出细粒度驱动的反哺式语义增强策略,对元数据进

行语义增强,构建领域感知的语义约束体系。

**5**)模糊需求数据集构建及自动化评估方法:构建 Bird-fuzzy 评估数据集(涉及 6 个领域),为面向模糊 需求场景的系统评估提供了数据支持。同时,我们在 该数据集上实现了针对模糊需求的自动化评估。

实验结果表明,SQL-MARS 有效支持用户完成从 模糊需求到精确查询的渐进式导航过程,为模糊需求 应用场景提供了解决方案。

# 2 相关工作

# 2.1 基于 LLM 的智能体研究

随着 LLM的兴起,基于 LLM的智能体潜力不断被 探索。调用工具是 LLM 智能体的关键能力,它们弥合 了 LLM 智能体与外部世界之间的差距。AutoGen[3]是 AI 智能体的开源框架,允许开发人员通过多个智能体 构建 LLM 应用程序,智能体之间可以相互通信完成任 务,同时配备了许多有用的工具以增强单个智能体的 能力。MetaGPT[4]开创性地将人类工作流程建模为元编 程框架,通过标准化操作程序(SOP)的提示编码实现 多智能体的结构化协作。XAgent[5]是一个开源的大 语言模型驱动的智能体,由 Dispatcher、Planner 和参与 者组成,分别负责任务分派、计划生成和执行,在处 理复杂指令方面的适应性、效率和精确性方面表现优 秀。在自然语言与数据库交互任务中,MAC-SQL[6]和 CHESS[7]使用多智能体框架,其中专用智能体分别负 责模式选择、分解和优化。使用多智能体可以将特定 任务分配给特定的智能体处理,各智能体之间协调工 作,能够提升整个系统的稳定性。本文同样采用多智 能体的框架实现面向模糊需求的感知与导航功能。

为提升基于 LLM 的智能体的推理与规划能力,研 究者提出了 ReAct 机制[8],通过 LLM 根据当前观察生 成思考与行动的迭代过程直至任务完成;Reflexion[9]则 引入自反思机制以优化推理质量。本文系统在这些机 制的基础上,采用元数据增强、基于 LLM 的自我反思 评估、优秀示例库实现推理的优化。

## 2.2 LLM 和数据库的关系

大型语言模型(LLMs)凭借其海量预训练数据和强 大的语义理解能力,正在重塑数据管理范式,推动下 一代智能数据库系统的发展。当前 LLM 在数据库领域 的应用已覆盖数据清洗[10]、查询优化[11]、SQL 等价性 判定[12]及故障根因分析[13]等核心场景。

针对 LLM 在数据库交互中的关键挑战,现有研究 提出了多种解决方案:DB-GPT系统[11]通过提示生成优 化、模型微调和领域适配设计解决了提示工程难题和 物理特征(如数据分布)感知问题;基于 LLM 增强的 数据分析平台 Chat2Data[1]通过提取领域数据生成语义 向量嵌入,利用向量数据库实现自然语言查询的智能 检索与响应。实现了低代码交互式数据分析,有效克 服了领域知识缺失和交互效率瓶颈。

SQL-MARS 在现有系统基础上进行了功能拓展: 其创新性的分层元数据导航机制支持从模糊需求到明 确需求的渐进式澄清过程,实现了更全面的智能交互 能力。

# 2.3 基于 LLM 的 Text-to-SQL 研究进展

大语言模型(LLM)凭借其强大的自然语言理解 与生成能力,已成为 Text-to-SQL 任务的核心技术范式。 现有研究主要围绕提示工程和模型微调两大方向展开 系统优化,其技术演进可划分为三个关键阶段:

- 1)基础提示工程阶段:早期研究聚焦于提示模板 的优化设计,包括零样本场景下的有效提示构造[14,15] 以及少样本学习中的示例选择策略[16]。这类方法通过 精心设计的指令模板(如"请将以下自然语言转换为 SQL查询")激活LLM的潜在能力,但受限于模型的固 有能力天花板,在复杂查询场景下表现有限。
- 2)模块化协作阶段:为突破单一提示的局限性, 研究者提出多模块协作策略。C3[17]开创性地将任务分 解为模式链接、SQL 生成和 SQL 验证三个子模块,显 著提升生成准确性。后续工作沿此方向持续改进: DIN-SQL[18]引入自修正机制处理语法错误,DEA-SQL[19]通过自主学习模块实现动态优化。这一阶段同

时关注模式信息的有效利用,如 ReFoRCE[20]采用表压 缩技术解决长上下文问题,MAC-SQL[6]则通过隐式模 式分解提供程序性知识指导。

3)推理增强阶段:随着思维链(CoT)技术的兴 起,研究者将其适配至 SQL 生成场景,但标准 CoT 提 示效果有限。QDecomp[21]创新性地按执行流将 SQL 子 句重组为逻辑步骤,相比通用 CoT 减少了错误传播。 CHASE-SQL[7]进一步提出"分治"策略,通过多路径推 理生成候选 SQL 并择优执行,效果得到了提升。

本系统在引导用户澄清数据需求后,需要借助 Text-to-SQL 技术将自然语言需求转化为数据库可执行 的 SQL 标准化语言并执行。过程中我们将增强的元数 据纳入提示词,设计具有领域约束的提示词,提升 SQL 生成质量。

### 2.4 对话式数据探索与自动化推荐问题生成

Maria 等人[22]关注多维度数据集的对话数据探索, 提出了一种多维数据模型,将数据组织成立方体,包 括维度,度量值,筛选器,通过交互确定三个维度的 数据子集,并基于数据库模式描述生成自然语言建议, 降低用户认知负荷。但是其对于用户查询使用正则表 达式提取实体,元数据匹配采用精确键值匹配,容易 丢失用户语义,泛化能力不足。

Ritwik 等人[23]聚焦于表格数据的自动化问题生成, 在对话式环境中自动基于数据集的相关切片推荐有趣 问题,并评估数据集切片的兴趣度得分来推荐高分问 题。但是子数据集的相关切片计算复杂度高,仅适用 于少量数据集的开放式数据探索,并且问题生成采用 的槽填充技术局限性大,难以广泛的在真实工业场景 的复杂需求中应用。

QUIS[24]关注于自动化数据探索(ADE)与洞察生 成,能自动生成特定数据集的问题并给出数据洞察的 可视化结果。该系统迭代式的问题生成策略不依赖于 初始 fewshot 示例,能够从数据中挖掘更广泛的洞察范 围,但是迭代计算时间长,不适用于用户有原始需求 和外部资料时快速数据分析的场景。

上述研究均致力于结构化数据集的开放性探索与 分析[22-24]。相较于本文提出的 SQL-MARS,CLARE[22] 严重依赖正则表达式和精确键值匹配,在处理自然语 言表达的模糊需求时泛化能力不足,且无法利用外部 资料;QUIS[24]侧重于无目标的数据探索和洞察发现, 其迭代式生成策略计算成本高,且不适用于用户已有 初始模糊意图并希望快速得到结果的场景。

# 3.1 框架设计

如图 1所示,对于用户的数据库查询需求,多智能 体框架的运行流程如下:首先进入①感知阶段, Planner 将给出查询需求模糊度的判定结果。然后进入 ②行动阶段,该阶段根据 Planner 的判定结果选择处理 通道,若判定为模糊需求则路由至建议反馈通道,交 由Advisor处理,Advisor将采用三级渐进式查询建议推 荐策略,通过多轮交互帮助用户实现需求精准化,对 于每一轮用户修改的需求都需要返回 Planner 再次判定 需求是否明确;若判定为明确需求,则路由至数据检 索通道,交由Seeker处理,Seeker将根据需求生成SQL 并查询出最终结果。最后进入③评估阶段,Evaluator 将根据整个交互过程的信息对系统性能进行评分。下 面 3.1.1 至 3.1.4 小节将详细介绍这四大智能体的功能以 及实现细节。

# 3.1.1 Planner(感知阶段)

**Planner** 作为多智能体框架的核心调度枢纽,负责 模糊度判定。其输入为用户的自然语言查询 Q,采用 分层元数据识别机制(详见第 4章)进行模糊度感知。 它通过 LLM 对 Q 进行语义解析,将其映射到数据库的 三级元数据(库、表或字段)。除了检查是否包含完 整的"库-表-字段"路径外,Planner 还引入了基于语义相 似度得分的阈值判定机制:若某一级(如表级或字段 级)存在多个高相似度候选(相似度>0.8),则即使路 径完整也判定为模糊需求,需进一步澄清。此外, Planner 还会统计歧义实体的数量,若某一层及候选数 超过预设阈值(设为 3),则同样判定为模糊需求。判 定结果将直接决定系统进入建议反馈通道或数据检索 通道。因此,Planner 通过模糊感知实现了处理路径的 动态调度,保证整个多智能体框架的自适应性。

### 3.1.2 Advisor(行动阶段-建议反馈通道)

**Advisor** 专用于处理模糊需求,通过多粒度元数据 检索与分层查询建议推荐引导用户逐步明确查询意图。 它接收来自 Planner 的模糊查询 Q,并启动分层引导流 程。其内部集成了向量检索模块,在库、表、字段三 个粒度上进行语义相似性搜索,获取 top-k 个高匹配度 元数据项。同时,Advisor 配备了一个外部文档融合工 具(详见第 6章),当用户上传外部资料时,该工具会 启动文档解析与指标抽取流程,从文档中识别出关键 统计指标及其业务描述。这些提取出的指标信息将与 元数据检索结果一同作为上下文提供给 LLM,增强其 对用户潜在查询意图的理解,从而使得生成的推荐意 见更贴合用户的业务场景。

整个推荐过程采用三级渐进式策略逐步实现需求 精准化:首先在库级层面识别潜在数据域(如将模糊 查询"学生数据"映射至教育库)帮助用户锁定查询领 域;随后根据用户选择在表级推荐相关实体(如学籍 表或成绩表)帮助用户锁定查询对象;最后在字段级 明确具体分析维度(如 GPA 或就业率)帮助用户锁定 查询指标。其中每一层的候选推荐均需满足跨层级一 致性校验(如仅保留"教育库→学籍表→GPA"的完整路 径),并充分融合从外部文档中提取的指标线索,结 合LLM生成满足当前场景的自然语言建议(如"您是否 需要查询数据学院本科生的就业率?"),生成三级建 议的 prompt 详见 github 项目仓<sup>1</sup>。用户选择推荐的某个 建议之后再次送至 Planner 进行新一轮模糊度感知,如 此循环往复,通过多轮交互逐步收敛至明确需求。

# 3.1.3 Seeker(行动阶段-数据检索通道)

**Seeker** 专用于处理明确需求,实施增强型检索 **SQL** 生成流程。其输入为用户已明确的自然语言需求 Q。Seeker 首先对 Q 进行语义解构,识别其操作类型、 目标实体及约束条件。随后,它在语义增强的元数据 向量库(详见第 5 章)和优质查询示例库(RAG 架构) 中联合检索,检索结果与用户查询 Q 共同构成提示词, 确保生成的 SQL 语句既符合语义意图又适配实际数据 库。生成的 SQL 会调用语法自检查工具进行规范性验

证。最终通过查询执行工具获取结果并反馈给用户。

## 3.1.4 Evaluator(评估阶段)

# **Evaluator** 实 现 了 多 维 度 的 系 统 性 能 评 估 。

Evaluator 会记录下交互过程中的会话交互轮次、LLM 响应时间两个客观指标,并结合 LLM 对上下文合理性 的自动评分(语义连贯性、需求满足度等)及用户显 式反馈(评分)两个主观指标,通过加权计算获得综 合评分。评分高于阈值(80 分)的对话会被向量化后 存入知识库中,形成关键知识沉淀:一方面作为建议 反馈通道的优质示例库,通过相似性检索提升后续意 图预测的准确度;另一方面作为数据检索通道的语法 模式库,减少明确需求下的 SQL 生成错误。这一阶段 不仅实现了系统表现的量化评估,更通过持续的知识 沉淀形成整个导航框架的自我优化能力。

# 3.2 关键技术挑战

整个多智能体协同数据导航框架在实现过程中面临 以下三个核心挑战,这些挑战直接影响模糊查询需求场 景下引导的有效性(能够渐进引导用户澄清需求)与生成 SQL 的可靠性(生成的 SQL 在数据库中可执行):

# **1**)自然语言模糊需求与数据库结构化数据映射问题

用户的模糊需求是以自然语言形式表达的,而数 据库中存储的是结构化数据,两者之间存在语义鸿沟, 无法进行有效映射,所以无法进一步通过判断需求的 元数据路径完整性来实现模糊度感知与渐进式引导。

# **2**)语义歧义导致 **SQL** 生成的幻觉问题

在跨领域数据中,即使用户明确自己的数据需求, 由于相同实体在不同业务场景下存在多义性冲突(如 "表现"在教育领域指"成绩",在人力资源领域对应"业 绩"),基于向量相似度的元数据检索面临语义歧义问 题,无法准确定位到数据库中的目标"库-表-字段"序列, 导致生成看似语法正确但语义失准的无效 SQL。

# **3**)外部资料与本地元数据的融合问题

用户上传的外部文档(如分析报告)中使用的业务 术语与本地数据库元数据之间往往存在三重差异:

A.主体差异:外部资料描述的实体(如"软件学院 研究生")在本地数据库(仅有"数据学院本科生")中 无直接对应;

B.术语差异:外部资料使用业务术语,而数据库多 采用英文编码字段。

C.粒度差异:外部指标通常为复合计算(如"就业 率"),而数据库仅存储基础字段。

上述挑战直接影响多智能体系统的实用性与可靠 性。本文后续章节将分别针对这些挑战提出解决方案, 并与框架中的智能体功能紧密对应:

针对挑战 **1**,第 **4** 章提出分层元数据架构。将为 Planner 的模糊度判定和 Advisor 的分层推荐提供基础的 语义映射支持,是实现渐进式引导的前提。

针对挑战 **2**,第 **5** 章提出元数据语义增强方法。将 直接应用于 Seeker 的检索与 SQL 生成流程,通过引入 领域约束抑制幻觉,确保生成 SQL 的语义准确性。

针对挑战 **3**,第 **6** 章提出外部文档解析与融合机制。 将作为 Advisor 的辅助工具,为其提供额外的意图线索, 增强推荐建议的精准性和上下文相关性。

# 4 基于分层元数据的查询需求澄清方法

如第 3.2 节所述,自然语言查询需求与结构化数据 之间存在语义鸿沟(挑战 1),这是实现模糊感知与渐 进式引导的首要障碍。本章旨在应对这一挑战,为第 3 章中所述 Planner 的模糊判定功能以及 Advisor 的分层 查询建议推荐功能提供核心技术支撑。

该挑战的根源在于两者在数据组织方式和表达形 式上存在异构性。数据库中的结构化数据可以天然地 抽象出"库-表-字段"三个粒度级别。通过检测需求的语 义解析结果能否映射出完整的"库-表-字段"访问路径, 我们可以有效评估查询需求模糊度,并据此逐步引导 用户澄清其数据需求。基于这一思路,本章提出了一 种基于三级元数据抽象的模糊需求建模与渐进式澄清 方法。下文将详细介绍分层元数据架构的具体实现。

### 4.1 分层元数据抽象架构

分层元数据的目标是将数据库元数据抽象为三个 层级结构:

- 1) 库级(Database Level): 标识完整的数据库实例,包括数据库名称、包含的表列表等全局信息。
- 2) 表级(Table Level): 描述数据库中的各个数据表,包括表名、表包含的所有字段等信息。
  - 3) 字段级(Column Level): 定义表中各个字段

的详细信息,包括字段名、数据类型、字段示例 (每个字段取出数据库中的3个真实值)等信息。

#### 4.4 渐进式澄清方法

基于三层元数据可以通过结构化匹配解决模糊需 求的判别问题。对于模糊需求的澄清问题,我们可以 对表征好的分层元数据进行语义相似性搜索来解决。 当用户提交查询需求时,系统将其语义解析,判断是 否已明确"库-表-字段"完整路径;若未明确则通过向量 相似度计算(similarity\_search)搜索各层级 top-k 个高 匹配度元数据,基于搜索到的元数据以及上下文信息 组织多粒度需求建议,在三个层级分别推荐。用户可 通过选择最匹配的推荐项逐步明确查询意图,完成需 求澄清。

# 5 基于元数据语义增强的 SQL 生成抗幻觉优化 法

如第 3.2 节所述,语义歧义问题(挑战 2)易导致 SQL 生成过程中出现幻觉现象,严重影响 Seeker 在数 据检索通道中的输出可靠性。本章旨在应对这一挑战, 通过为元数据注入丰富的语义信息,提升 Seeker 生成 SQL 的语义准确性,从而有效缓解跨领域查询中的多 义性冲突问题。

在跨领域数据库的自然语言查询场景中,即使用 户明确表达了数据需求,仍然可能因为实体多义性导 致语义歧义问题。以"表现"一词为例,在教育领域常 被解释为学生的成绩,而在人力资源管理场景中通常 指员工的绩效指标。传统向量匹配方法由于缺乏深度 的上下文建模能力,难以准确识别领域特定的语义指 向,导致生成的 SQL 查询虽然语法正确但语义失准, 严重影响了查询结果的业务相关性。

为了解决上述问题,本章节提出一种基于元数据 语义增强的歧义消除方法。整体结构如图 3 所示,一方 面,将"自上而下"的分层元数据表征信息纳入提示词, 使 LLM 能够更好地理解语义;另一方面,实施反哺式语 义增强,通过"自底向上"的信息传递路径,在第 4 章 获得字段级精细语义后,由字段级信息增强表级语义, 再由表级语义反向增强数据库级语义:

- **1**)字段信息增强表信息:通过字段之间的上下文 联动与语义归一,提升表的语义清晰度与一致性。结 合字段表征时生成的字段描述、关联分析结果、备选 描述、字段类型,重构表的描述、语义场景等以达到 增强表信息的目的。
  - **2**)表信息增强库信息:基于表间语义聚合,推断

数据库的核心业务语义,基于此修改数据库的描述, 完成增强数据库信息构建。

最终生成的增强元数据包含领域标记、场景标签 和语义约束三重信息,提升了数据库语义空间的一致 性与判别能力,为查询推荐提供了场景指向,为后续 的自然语言到 SQL 映射提供了更高质量的语义基础, 有效抑制因语义歧义导致的查询幻觉现象。

# 6 融合外部资料与本地数据的数据探索问题推 荐机制

如第 3.2 章所述,外部资料与本地元数据在主题、 术语和粒度上均存在差异(挑战 3),为查询意图的捕 捉带来了额外困难。本章旨在应对该挑战,通过构建 一个融合外部资料与本地元数据的语义桥梁,扩展 Advisor的上下文感知与推荐能力。它作为Advisor的一 个辅助工具被调用,能够在用户上传外部文档后自动 解析文本、提取关键统计指标,并生成与本地数据库 语义一致的高质量查询建议,从而更高效地引导用户 澄清模糊的数据需求。

在真实的数据探索场景中,用户提供的报告、文 档等外部资料往往蕴含重要的查询意图线索。例如, 若用户上传一份《软件学院招生数据分析报告》,即 使本地数据库中仅存有"数据学院就业数据",系统仍 可借助外部语义推断并推荐如"统计数据学院招生情况" 等潜在查询方向。这种跨源语义融合能力显著提升了 模糊需求的识别与澄清效率。

本章提出的融合外部资料与本地数据的数据探索 问题推荐机制通过构建完整的处理链路(外部资料解 析与统计指标实体抽取→元数据验证与指标选择→智 能问题生成与推荐)实现该功能。我们充分认识到, 从非结构化文本中自动化、高精度地抽取指标公式涉 及到对自然语言的深层理解和数学逻辑解析,本系统 未追求完全通用的公式抽取,而是采用宽松抽取与严 格验证相结合的方式提高最终推荐问题的质量。下文 将详细介绍该融合机制的实现流程。

#### 6.1 外部资料解析与指标实体识别

基于 LangChain 的文档处理流水线,采用中/英文 友好的文档分块方式将外部资料(PDF 格式)切分为 语义连续的文档块。随后,指标抽取器会识别文本中 的统计指标(如"就业率"、"毕业生薪资","毕 业生总人数"),对于一些需要聚合等操作的统计指 标(如"就业率=就业人数/毕业生总人数×100%"), 解析出相应的数学计算公式,并且对于公式中涉及的 每个字段生成业务场景下的语义描述。

指标抽取器通过文本块,以及优质示例库中的数 据指标抽取实例填充到统计指标抽取的提示词模板中 (具体的 prompt 详见 github 项目仓库),使用 LLM 抽 取文档块中的统计指标实体。其中最核心的是对于指 标计算公式的解析,提示词的系统指令要求 LLM 使用 "SQL 语法结构",并且将公式中涉及的过滤条件放 入 where 子句中。对于最值、均值、计数、求和字段使 用聚合函数实现,避免 LLM 使用预设字段导致幻觉, 旨在提升指标公式抽取的准确性。指标抽取器调用 LLM 进行指标抽取后,将返回结果解析为指标抽取列 表存入本地缓存,用于后续元数据匹配和问题推荐, 同时避免重复文本块对 LLM 的多次调用。

通过指标抽取模板设计提高文本块指标抽取的召回率, 充分挖掘外部资料包含的指标,并由后续的"元数据验证 与指标选择"环节充当严格的过滤器,实现"宽松抽取"与 "严格验证"相结合的模式提升最终推荐问题的质量。

#### 6.2 元数据验证与指标选择

在语义对齐阶段,如图 4所示,使用 BGE-M3嵌入 模型将外部资料中抽取到的指标对应的描述与本地数 据库的列级元数据同时映射到统一的语义空间,计算 两者的向量相似度;然后设置动态匹配阈值进行指标 选择。考虑到统计指标的复杂性,一个有意义的统计 指标可能是对数据库中某个数据子集基于列值或者时 间序列的综合分析,因此将指标计算需要的字段与列 级元数据匹配的结果会出现多种情形(如"部分匹配"、 "完全匹配"),本方法中仅当指标公式中所有字段都 满足阈值要求时,即"完全匹配"时判定该统计指标可 以在本地数据库查询并选中该指标,反之丢弃该指标。 此外,需要注意的是,匹配阈值可以根据不同应用场 景进行调整,对于金融,医疗等精细化要求高的领域 可以设定为 0.9 及以上。

#### 6.3 智能问题生成与推荐

对于通过验证的统计指标,整合匹配字段的元数 据信息(元数据描述等)和原始文本块(含上下文业 务信息),将其作为输入提供给 LLM,生成符合领域 场景的数据分析问题,作为查询意图参考传递给 Advisor 智能体,辅助其生成符合用户认知习惯的自然 语言查询建议。

# 7 案例分析

本章节通过具体案例,将本文提出的 SQL-MARS 系统与基线方法(Baseline)进行对比分析,以凸显 SQL-MARS 在处理模糊查询需求时的有效性与优势。

# 7.1 基线方法介绍

我们选择的基线采用零样本、单轮的查询方式。将 用户的自然语言查询与未经分层化处理的混合的数据库 元数据(即所有数据库的 schema 拼接成一个长文本) 一次性输入到大语言模型中,直接利用 LLM 的理解和 生成能力将自然语言转换为SQL查询语句,不含交互澄 清机制。为实现公平对比,我们使用与 SQL-MARS 系 统相同的 LLM 作为基线方法的推理引擎。我们选择此 方法作为 Baseline,旨在清晰地揭示:若缺乏针对模糊 查询的专门设计,现有系统将面临的根本性困境。

# 7.2 教育领域场景对比分析

# **1**)**SQL-MARS**

交互过程如表 3 所示。

第一轮,用户给出的查询"学生信息"判定为模糊 查询。由于数据集跨六大领域,有多个领域涉及"学生 信息",所以系统会搜索分层元数据库,分别从库、表、 字段级猜测用户查询意图。库级层面聚焦"学生社团成 员信息"、"加利福尼亚学生信息"、"学生银行账户及贷 款 信 息", 分别对应数据集中的 student\_club, california\_schools,financial 数据库,横跨 EDU 和 FIN 两大领域。相应的,表级层面关联到具体的表格给出 具体事件的查询推荐,字段级层面给出具体属性的查 询推荐。

第二轮,用户选择库级推荐建议"加利福尼亚州学 生信息"。系统会识别到库级需求已明确,则反馈的表 级、字段级查询建议会限定至 california\_schools 库以提 升推荐质量。

第三轮,用户没有选择建议,而是上传文档。如 图 5 所示,系统首先会执行提取文档指标("SAT 优秀 率"等)、映射本地数据库字段(如 NumGE1500→SAT 高分人数)两大步骤。然后判断指标与当前上下文用 户的模糊需求("加利福尼亚学生信息")相符,进而 生成对应的字段级查询建议。

用户最终选择查询"哪些学校的学生 SAT 优秀率超 过 90%",此时系统判定需求明确,生成对应的 SQL 以 及解释,并将查询结果反馈给用户。该 SQL 与用户的 真实意图相符且可执行。

#### **2**)**Baseline**

最终生成的 SQL 既不满足用户查询意图,也无法 执行出结果。由生成的 SQL,我们可以发现查询的是 student\_club 库的事件信息,与用户的意图("学生信 息")不符。

此案例典型地揭示了基线方法在模糊查询场景下 的固有缺陷。一方面,大语言模型从模糊需求中获得 的信息有限,由于 student\_club 和 california\_schools 数 据库都涉及到"学生信息",导致使用另一数据库领域 的元数据回答问题,出现答非所问现象。另一方面, 由于查询的时候没有提供增强的元数据(特别是字段 示例值),LLM 过度依赖语义联想,此处表现为 "status = 'active'",数据库中的 status 字段只存在 Open, Closed,Planning 三个可能属性值。

# 7.3 金融领域场景

#### **1**)**SQL-MARS**

交互过程如表 4 所示。

第一轮,用户给出的查询"查询用户消费情况"判 定为模糊查询。由于"消费"一词在不同领域表述不同, 所以系统会将涉及到该关键词的领域都推荐给用户, 方便用户首先锁定自己的查询领域。库级层面聚焦"信 用卡交易消费"、"客户订单消费"、"学生社团财务", 分别对应数据集中的 debit\_card\_specializing ,financial , student\_club 数据库,横跨 EDU 和 FIN 两大领域。表级 层面、字段级层面给出的推荐与上述的领域保持一致。

第二轮,用户选择库级推荐建议"客户的消费记录"。 系统会识别到库级需求已明确,则反馈的表级、字段级 查询建议会限定到 debit\_card\_specializing数据库以提升 推荐质量。为帮助用户建立对数据库的初步认知,系 统会在建议中附带相关的元数据信息。

第三轮,用户选择建议"查询 yearmonth 表中特定 客户的每月消费详情"。可见用户需求已明确至具体数 据表,因此系统将聚焦于该表的字段属性值与聚合指 标,并据此提供查询建议。

第四轮,用户选择查询"找出 2013 年消费最多的客 户",系统识别查询需求已明确,系统生成 SQL 并将查 询结果返回给用户。至此,交互结束,整个过程中, 系统在逐级引导用户明确需求,遵循"锁定领域—锁定 事件—锁定指标"的渐进式澄清链路,保证最终生成高 质量的 SQL。

#### **2**)**Baseline**

此场景下生成的 SQL 可执行但与用户意图不符。 同样存在与教育场景相似的问题,这是因为 financial和 debit\_card\_specializing 数据库属于同一领域,都涉及到 "消费"的信息,而大语言模型从模糊需求中获得的信 息有限,且无法通过与用户交互对同一领域内的相似 子库进行选择。进一步验证了基线方法的局限性:不 具备与用户交互的能力,完全依赖 LLM 自身的能力从 模糊需求中猜测用户意图。

#### 7.4 小结

综上,由上述两个场景案例可见,Baseline 存在以

下两个共性问题:

- 1)多领域歧义问题:模糊查询中的关键词在不同 领域中表述不同,往往涉及多个数据库,因模糊需求 中的信息有限且无法与用户交互,出现答非所问现象。
- 2)元数据幻觉问题:LLM 因缺乏字段级示例等增 强数据,导致过度依赖语义联想,生成的 SQL 存在幻 觉现象。

相比之下,本研究提出的 SQL-MARS 方法可以先 推荐粗粒度的需求帮助用户锁定领域,然后再渐进式 推荐细粒度的查询建议,逐渐帮助用户锁定查询对象、 查询指标,可以有效避免上述基线方法存在的问题。 可见 SQL-MARS 在模糊需求场景下的有效性,以及面 向模糊需求的数据导航的必要性。

#### **2**)模糊化规则应用

术语层保留领域主体词(如教育领域的"学生","机 构","分数"等),替换专业术语为上位词(如医疗领域 "血红蛋白"转换为"血液指标");条件层隐藏具体数值/ 时间(如">180mg/dL"→"异常高值")、模糊化地点约 束(如"加利福尼亚州"→"某些地区"),操作层将显示 的聚合函数模糊描述为相关信息(如"AVG(age)"→"统计 年龄信息")。对于每个数据库生成的20个模糊需求进行 人工筛选,保留模糊度合理、与原始查询意图趋向一致 的高质量样本,最终 100 个模糊需求构成数据集,每个 模糊需求样本包含以下信息:问题 id,领域标签,数据 库名,语义线索,原始查询,模糊查询,真实 SQL。

#### 8.3.1 模型选择

为系统评估分层元数据架构在数据导航任务中的有 效性及泛化能力,本研究设计了多维度的模型对比实验。 基于模型规模与专业特性(指令遵循能力和工具调用能 力等)的差异化考量,我们选取了以下四个具有代表性 的开源大语言模型作为 SQL-MARS 系统的核心推理引 擎:LLaMA3.1-8B, Qwen2.5-7B, Qwen2.5-32B, Qwen2.5- Coder-32B。

# 8.3.3 基于综合性能的模型选择实验

表 7是在元数据分层情况下的各模型性能评估与平 均 token 数消耗情况。结果表明,模型表现与计算开销 呈现明显的正相关关系。Qwen2.5-32B 虽然取得了最高 的平均评分(81.39),但其 Token 消耗达到 13,536.56, 是 LLaMA3.1-8B 的 24 倍。由于 Qwen2.5-32B 的平均评 分显著领先(较次优模型提升 10.3%),且其平均 token 数在日常应用中是可接受的,所以综合考量下我 们选择 Qwen2.5-32B 模型作为 SQL-MARS 的推理模型。

# 8.3.4 实验局限性及改进方向

本次实验所有智能体均采用统一的模型,未能充 分考虑不同子任务的特性差异。同时,从不同领域的 外部资料中自动、准确地抽取"指标公式"是一项极 具挑战性的任务,其准确率高度依赖于优秀指标实体 抽取实例、领域内高质量的本体资源、prompt 模板设 计以及大模型指令遵循能力,尽管本文提出的外部资 料融合机制在实验示例中取得了不错的效果,但是由 于缺少包含外部资料的 benchmark,本文并未设计实验 统计分析该机制的效果。此外,Bird-fuzzy 数据集规模 较小(仅 100条),且模糊化规则人为设定,可能未覆 盖所有真实模糊场景。后续研究可探索异构模型组合 策略、扩大数据集规模,并研究提高外部资料指标抽 取精度的方法,以提升系统泛化性能。

结束语 本文实现了 SQL-MARS,提出了基于多智 能体协同的数据导航框架,该框架专注于面向数据库 模糊查询需求的动态判别与自适应处理。我们提出元 数据分层的方法构建"库-表-字段"三级语义映射体系, 弥合自然语言与结构化数据之间的语义鸿沟。对于模 糊需求,我们设计了基于三层增强元数据检索、优秀 示例库检索的多粒度需求建议推荐,以渐进式引导用 户明确数据需求。此外,我们探索性地实现了外部非 结构化文档与本地结构化元数据的语义对齐,提取有 价值的指标以辅助需求建议推荐。这种融合外部语义 的推荐方式,为跨域数据探索提供了新的交互范式。 在 Bird-fuzzy数据集上的实验结果表明,SQL-MARS能 够识别模糊需求并有效引导用户澄清。后续将进一步 根据不同子任务的特性差异探索异构模型组合策略; 对于外部资料中抽取的指标可能与本地数据库数据部 分匹配的问题,如何在这类场景下合理地推荐数据分 析问题也需要进一步探索。
