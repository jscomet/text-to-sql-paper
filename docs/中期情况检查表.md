| **存在的问题** | 1、**评测效率与准确性的冲突**：在实验迭代中，简单的哈希值比对法虽然快，但无法处理"逻辑等价但写法不同"的 SQL，导致评测结果偏低；而全量执行验证在 Bird 等大数据集上耗时过长，严重拖慢了模型微调后的验证周期，急需一种兼顾速度与精度的评测方案。`<br>`2、**复杂数据库结构的理解偏差**：在处理 Bird 等包含大量表和模糊字段名（如缩写、专业术语）的数据库时，模型常出现"张冠李戴"的情况。单纯依靠 Prompt 传入 DDL 信息，模型仍会产生幻觉，生成不存在的列名或错误的外键关联，导致生成的 SQL 无法执行。`<br>`3、**系统集成中的响应超时与鲁棒性问题**：由于 14B 规模的模型推理耗时较长（通常需数秒），导致前端网页常因等待响应而出现假死或超时报错。此外，当模型生成语法错误的 SQL 时，系统往往直接返回后端报错代码，缺乏对非技术用户的友好提示和自动纠错能力，演示系统的稳定性有待加强。 |
| **拟采取的办法** | 1、**建立分级评测验证机制**：针对不同复杂度的查询，开发一套"哈希初步过滤 + 抽象语法树（AST）深度比对"的混合方案。对于简单查询利用哈希快速过检，对于复杂聚合查询则解析 SQL 的语法树结构，判断其逻辑是否与标准答案等价，从而在不增加过多耗时的前提下提升评测的公信力。`<br>`2、**优化 Schema 描述与检索策略**：引入 RAG（检索增强生成）思路，将数据库的元数据和字段注释存入向量库。在提问前，先检索出最相关的表结构片段，并在 Prompt 中加入 2-3 条真实的示例数据（Sample Rows）。这种"看样学样"的方式能有效帮助模型识别字段含义，纠正缩写歧义，降低映射错误率。`<br>`3、**采用轻量化前后端分离架构**：前端使用 Vue 实现交互界面，通过 Axios 异步调用基于 FastAPI 搭建的后端接口，解决推理耗时导致的界面阻塞问题。引入简单的异常截获逻辑，当 SQL 执行失败时，不再直接报错，而是将错误信息反馈给模型尝试"一次自修复"或向用户返回直观的错误提示；同时利用本地缓存减少重复查询的等待时间，确保演示流程顺畅。 |
| **指 导 教 师 建 议 与 意 见** | 1、**核心逻辑的验证深度**：在解决"多解 SQL"评测难点时，AST（语法树）比对虽然比哈希值更精准，但在处理复杂的 JOIN 关联顺序时仍可能有局限。建议在论文中增加一小组人工抽样比对实验，量化分析 AST 相比传统哈希比对在准确率上的具体提升，这能显著增强你实验结论的说服力。`<br><br>`2、**数据增强的针对性**：针对 Schema 映射偏移问题，引入 RAG 和 Sample Rows（示例行）是非常务实的做法。建议在后续微调 14B 模型时，专门构造一批包含"缩写字段"和"多表同名字段"的干扰样本进行强化训练。通过这种"错题集"式的微调，能有效观察模型在垂直领域下的鲁棒性变化，是论文中非常好的创新点分析。`<br><br>`3、**原型系统的交互闭环**：在前后端分离的开发中，不要仅满足于"输入自然语言-输出表格"。建议增加一个"执行逻辑解析"展示位，将 Agent 推理的中间过程（如：它识别出了哪些表、连接条件是什么）简要展示给用户。这不仅能降低由于模型推理慢带来的焦虑感，也能在 SQL 执行报错时，让用户直观看到是哪个环节出了问题，体现出系统设计的完整性。`<br><br>`4、**进度管理与文档规范**：目前中期进度符合要求，但后续涉及 14B 模型的微调与多轮评测，算力资源与时间成本较高。建议严格执行实验日志记录，保存好不同版本模型的对比参数，为最后的毕业论文提供详实的图表支撑。`<br><br><br>`签字：　　　　　　年　　月　　日 |
