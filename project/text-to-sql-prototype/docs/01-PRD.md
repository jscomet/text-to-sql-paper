## 2. 业务背景

### 2.1 问题陈述

在现代企业中，数据库存储着大量业务数据，但非技术人员往往难以直接查询这些数据。传统的数据查询方式面临以下挑战：

1. **技术门槛高**：SQL 查询需要掌握特定的语法知识，业务人员难以快速上手
2. **效率低下**：简单的数据需求也需要依赖开发/数据团队，沟通成本高
3. **错误率高**：手写 SQL 容易出现语法错误，特别是在复杂的多表关联场景
4. **知识断层**：业务人员最懂业务需求，但不懂如何编写查询；技术人员懂 SQL，但不理解业务语境

### 2.2 解决方案

Text-to-SQL 智能查询系统利用大语言模型（LLM）技术，将自然语言自动转换为可执行的 SQL 查询，让非技术人员也能直接查询数据库，降低数据获取门槛，提升业务决策效率。

### 2.3 市场机会

- 数据分析市场规模持续增长，自助式 BI 需求旺盛
- LLM 技术成熟，Text-to-SQL 准确率已达到实用水平（BIRD 榜单 SOTA 超过 70%）
- 企业内部存在大量未被满足的数据查询需求

## 3. 目标用户

### 3.1 用户角色画像

#### 角色一：数据分析师
- **特征**：具备一定 SQL 基础，但不想记忆复杂的表结构和字段名
- **痛点**：
  - 需要频繁查询不同业务线的数据，每次都要查阅文档
  - 复杂的多表 JOIN 容易写错
  - 希望快速验证数据假设
- **需求**：通过自然语言快速生成 SQL，作为查询起点进行微调

#### 角色二：业务人员（运营/产品/销售）
- **特征**：不懂 SQL，但需要经常查看业务数据
- **痛点**：
  - 每次查数据都要找开发提需求，等待时间长
  - 无法自主探索数据，影响决策效率
  - 需求描述和实际结果经常存在偏差
- **需求**：零代码查询数据，通过自然语言直接获取结果

#### 角色三：开发者
- **特征**：熟悉 SQL，但需要快速生成模板或验证逻辑
- **痛点**：
  - 重复编写类似的查询语句
  - 需要快速理解不熟悉的业务数据库结构
  - 手写复杂查询耗时且容易出错
- **需求**：快速生成 SQL 模板，支持编辑和优化

#### 角色四：AI 研究人员
- **特征**：从事 Text-to-SQL 模型研究与评测
- **痛点**：
  - 缺乏标准化的评测工具和流程
  - 难以对比不同模型的性能
  - 需要可视化的评测结果分析
- **需求**：完整的评测 Pipeline，支持多种评估指标和可视化报告

### 3.2 用户场景总结

|------|------|------|
| 日常数据查询 | 业务人员 | 快速获取业务数据，无需技术依赖 |
| SQL 辅助编写 | 数据分析师/开发者 | 提高效率，减少错误 |

## 4. 产品目标

### 4.1 产品愿景

打造一款易用、准确的 Text-to-SQL 智能查询工具，让每个人都能通过自然语言轻松查询数据库，释放数据价值。

### 4.2 MVP 阶段目标

在 MVP 阶段，产品需要实现以下核心能力：

1. **自然语言转 SQL**：用户输入中文/英文自然语言，系统生成可执行的 SQL
2. **多数据源支持**：支持主流数据库（MySQL、PostgreSQL、SQLite）
3. **SQL 执行与结果展示**：执行生成的 SQL，返回查询结果
4. **评测体系**：支持标准数据集（BIRD）评测，输出准确率报告

### 4.3 成功标准

| 指标 | 目标值 | 说明 |
|------|--------|------|
| SQL 生成准确率 (EX) | >= 60% | 在 BIRD-dev 数据集上执行准确率 |
| 查询响应时间 | < 5s | 从输入到返回结果的平均时间 |
| 用户满意度 | >= 4.0/5.0 | 内部试用反馈评分 |
| 系统可用性 | >= 99% | 服务正常运行时间 |

## 5. 高级推理功能 (v1.1.0)

### 5.1 功能背景

当前 Text-to-SQL 系统采用 **Greedy Search**（贪婪搜索）模式进行 SQL 生成，该模式虽然响应快速，但存在以下局限：

1. **准确率受限**：单次采样容易受模型随机性影响，无法充分发挥模型的最佳潜力
2. **缺乏自我修正能力**：生成错误 SQL 后无法自动识别和修复，需要人工介入
3. **评估维度单一**：仅用 Greedy Search 评估会低估模型真实能力，无法全面反映模型潜力

**重要概念区分**：

| 概念层级 | 说明 | 具体手段/策略 |
|----------|------|---------------|
| **推理手段** | 如何生成 SQL | Single/Greedy、Sampling、Check-Correct |
| **评测手段** | 如何评估结果 | Greedy Search、Pass@K、Majority Voting |
| **组合策略** | 推理+评测的完整方案 | vote@k、pass@k |

**组合策略说明**：
- `vote@k` = Sampling (推理，n=K) + Majority Voting (评测)
- `pass@k` = Sampling (推理，n=K) + Pass@K (评测)

本功能将上述先进组合策略产品化，为用户提供 vote@k、pass@k 和 Check-Correct 等高级推理能力。

### 5.2 目标用户

| 用户角色 | 核心需求 | 使用场景 |
|----------|----------|----------|
| **评测人员** | 全面评估模型性能 | 使用 Pass@K 模式评估模型潜力上限 |
| **开发者** | 提高 SQL 生成准确率 | 使用 Check-Correct 模式获得更可靠的 SQL |
| **研究人员** | 对比不同推理策略效果 | 对比 Greedy/Pass@K/Check-Correct 的效果差异 |
| **业务用户** | 获取准确的查询结果 | 在重要场景下使用高级推理模式 |

#### 5.3.2 Check-Correct 迭代修正

**流程架构**：

```
初始化: 用户问题 + Schema 信息
    │
    ▼
┌──────────┐
│ Generator│ 生成初始 SQL (SQL_0)
└────┬─────┘
     │
     ▼
┌──────────┐
│ Executor │ 执行 SQL_0
└────┬─────┘
     │
     ▼
┌──────────┐      通过      ┌──────────┐
│ Checker  │ ─────────────▶ │ 返回结果 │
│ 检查正确性│                └──────────┘
└────┬─────┘
     │ 未通过
     ▼
┌──────────┐
│Corrector │ 生成修正 SQL (SQL_1)
└────┬─────┘
     │
     └──────────────▶ (进入下一轮循环)

终止条件:
- Checker 返回通过
- 达到最大迭代次数
- 超时
```

| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 初始生成 | Generator 生成初始 SQL | P0 |
| 执行验证 | 执行 SQL 获取结果或错误 | P0 |
| 正确性检查 | Checker 判断 SQL 是否正确 | P0 |
| 错误修正 | Corrector 根据错误生成修正 SQL | P0 |
| 迭代控制 | 管理迭代流程和状态 | P0 |

**迭代配置**：

| 配置项 | 说明 | 默认值 | 范围 |
|--------|------|--------|------|
| max_iterations | 最大迭代次数 | 3 | 1-5 |
| timeout | 单次迭代超时 | 30s | 10-60s |
| early_stop | 无改进时提前停止 | true | true/false |

**错误分类**：

| 错误类型 | 说明 | 示例 | 修正策略 |
|----------|------|------|----------|
| **Syntax** | SQL 语法错误 | 缺少括号、关键字拼写错误 | 语法修正 Prompt |
| **Execution** | 执行时错误 | 表不存在、字段不存在 | 结合错误信息修正 |
| **Semantic** | 语义错误 | 逻辑错误、条件错误 | 意图理解修正 |

| 平票处理 | 平票时选择第一个或随机选择 | P1 |
| 置信度评分 | 基于投票比例和执行成功率计算置信度 | P1 |

**全失败处理策略**：

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **返回空** | 返回空结果或错误提示 | 严格场景 |
| **返回随机** | 随机返回一个候选 | 探索性场景 |
| **返回最后** | 返回最后一个候选（可能有部分结果） | 容错场景 |
| **触发修正** | 自动进入 Check-Correct 流程 | 高精度场景 |

#### 5.6.1 术语表

| 术语 | 英文 | 说明 |

| pass@k | Pass at K | K次采样 + 任一通过的组合策略 |

| **评测手段** | **Evaluation Method** | **如何评估结果的方式** |
| Pass@K | Pass at K (Metric) | K 次采样中至少有一次正确的概率 |

| Greedy Search | Greedy Search | 贪婪搜索，单次结果评测 |

#### 5.6.3 错误码定义

| 错误码 | 说明 | 处理建议 |
|--------|------|----------|
| 4001 | 不支持的推理模式 | 检查 inference_mode 参数 |
| 4002 | K 值超出范围 | K 应在 [1, 4, 8, 16] 中 |
| 4003 | 迭代次数超出范围 | max_iterations 应在 1-5 之间 |
| 5001 | 推理超时 | 增加超时时间或简化问题 |
| 5002 | 所有采样失败 | 检查模型配置或问题表述 |
| 5003 | 迭代未收敛 | 增加 max_iterations 或调整 temperature |
