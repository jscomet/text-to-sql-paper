## 2. 业务背景

### 2.1 问题陈述

在现代企业中，数据库存储着大量业务数据，但非技术人员往往难以直接查询这些数据。传统的数据查询方式面临以下挑战：

1. **技术门槛高**：SQL 查询需要掌握特定的语法知识，业务人员难以快速上手
2. **效率低下**：简单的数据需求也需要依赖开发/数据团队，沟通成本高
3. **错误率高**：手写 SQL 容易出现语法错误，特别是在复杂的多表关联场景
4. **知识断层**：业务人员最懂业务需求，但不懂如何编写查询；技术人员懂 SQL，但不理解业务语境

### 2.2 解决方案

Text-to-SQL 智能查询系统利用大语言模型（LLM）技术，将自然语言自动转换为可执行的 SQL 查询，让非技术人员也能直接查询数据库，降低数据获取门槛，提升业务决策效率。

### 2.3 市场机会

- 数据分析市场规模持续增长，自助式 BI 需求旺盛
- LLM 技术成熟，Text-to-SQL 准确率已达到实用水平（BIRD 榜单 SOTA 超过 70%）
- 企业内部存在大量未被满足的数据查询需求

## 3. 目标用户

### 3.1 用户角色画像

#### 角色一：数据分析师
- **特征**：具备一定 SQL 基础，但不想记忆复杂的表结构和字段名
- **痛点**：
  - 需要频繁查询不同业务线的数据，每次都要查阅文档
  - 复杂的多表 JOIN 容易写错
  - 希望快速验证数据假设
- **需求**：通过自然语言快速生成 SQL，作为查询起点进行微调

#### 角色二：业务人员（运营/产品/销售）
- **特征**：不懂 SQL，但需要经常查看业务数据
- **痛点**：
  - 每次查数据都要找开发提需求，等待时间长
  - 无法自主探索数据，影响决策效率
  - 需求描述和实际结果经常存在偏差
- **需求**：零代码查询数据，通过自然语言直接获取结果

#### 角色三：开发者
- **特征**：熟悉 SQL，但需要快速生成模板或验证逻辑
- **痛点**：
  - 重复编写类似的查询语句
  - 需要快速理解不熟悉的业务数据库结构
  - 手写复杂查询耗时且容易出错
- **需求**：快速生成 SQL 模板，支持编辑和优化

#### 角色四：AI 研究人员
- **特征**：从事 Text-to-SQL 模型研究与评测
- **痛点**：
  - 缺乏标准化的评测工具和流程
  - 难以对比不同模型的性能
  - 需要可视化的评测结果分析
- **需求**：完整的评测 Pipeline，支持多种评估指标和可视化报告

### 3.2 用户场景总结

| 场景 | 用户 | 目标 |
|------|------|------|
| 日常数据查询 | 业务人员 | 快速获取业务数据，无需技术依赖 |
| SQL 辅助编写 | 数据分析师/开发者 | 提高效率，减少错误 |
| 数据库探索 | 新加入的开发者 | 快速理解数据库结构 |
| 模型评测 | 研究人员 | 系统评估 Text-to-SQL 模型性能 |

## 4. 产品目标

### 4.1 产品愿景

打造一款易用、准确的 Text-to-SQL 智能查询工具，让每个人都能通过自然语言轻松查询数据库，释放数据价值。

### 4.2 MVP 阶段目标

在 MVP 阶段，产品需要实现以下核心能力：

1. **自然语言转 SQL**：用户输入中文/英文自然语言，系统生成可执行的 SQL
2. **多数据源支持**：支持主流数据库（MySQL、PostgreSQL、SQLite）
3. **SQL 执行与结果展示**：执行生成的 SQL，返回查询结果
4. **评测体系**：支持标准数据集（BIRD）评测，输出准确率报告

### 4.3 成功标准

| 指标 | 目标值 | 说明 |
|------|--------|------|
| SQL 生成准确率 (EX) | >= 60% | 在 BIRD-dev 数据集上执行准确率 |
| 查询响应时间 | < 5s | 从输入到返回结果的平均时间 |
| 用户满意度 | >= 4.0/5.0 | 内部试用反馈评分 |
| 系统可用性 | >= 99% | 服务正常运行时间 |

## 6. 非功能需求

### 6.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 页面加载时间 | < 2s | 首屏加载时间 |
| SQL 生成时间 | < 3s | 从输入到生成 SQL 的时间 |
| 查询执行超时 | 30s | 单条 SQL 最大执行时间 |
| 并发用户数 | >= 10 | MVP 阶段支持的同时在线用户数 |
| 评测任务并行 | >= 4 | 支持并行的评测进程数 |

### 6.2 安全需求

| 需求 | 说明 |
|------|------|
| SQL 注入防护 | 对用户输入进行过滤和验证 |
| 数据库权限隔离 | 使用只读账号连接生产数据库 |
| 敏感数据脱敏 | 对查询结果中的敏感字段进行脱敏处理 |
| 连接加密 | 数据库连接使用 SSL/TLS 加密 |

### 6.3 可用性需求

| 需求 | 说明 |
|------|------|
| 易用性 | 界面简洁直观，非技术人员无需培训即可使用 |
| 容错性 | 生成错误的 SQL 时给出友好提示，不直接暴露错误堆栈 |
| 可恢复性 | 评测任务支持断点续跑 |
| 浏览器兼容 | 支持 Chrome、Firefox、Edge 最新两个版本 |

### 6.4 可维护性需求

| 需求 | 说明 |
|------|------|
| 代码规范 | 遵循团队编码规范，关键代码有注释 |
| 日志记录 | 记录关键操作和错误日志，便于排查问题 |
| 配置化管理 | 数据库连接等配置支持热更新 |

## 7. 风险评估

### 7.1 技术风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| LLM 生成 SQL 准确率低 | 高 | 中 | 1. 使用 Schema 信息增强提示<br>2. 引入 few-shot 示例<br>3. 设置人工审核机制 |
| 复杂 SQL 生成失败 | 中 | 高 | 1. 分步骤生成（先子查询后组合）<br>2. 限制 MVP 阶段支持的 SQL 复杂度 |
| 大表查询性能问题 | 高 | 中 | 1. 默认添加 LIMIT 限制<br>2. 执行超时机制<br>3. 只读账号连接 |
| 数据库连接安全 | 高 | 低 | 1. 使用连接池<br>2. 密码加密存储<br>3. 最小权限原则 |

### 7.2 业务风险

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 用户不信任 AI 生成结果 | 中 | 中 | 1. 展示生成的 SQL 供用户确认<br>2. 提供执行结果预览<br>3. 支持手动编辑 SQL |
| 业务数据库 schema 变更 | 中 | 高 | 1. 定期同步 schema 信息<br>2. 版本化管理 schema |
| 数据隐私合规 | 高 | 低 | 1. 敏感数据脱敏<br>2. 查询日志审计<br>3. 用户权限隔离 |

## 8. 附录

### 8.1 术语表

| 术语 | 英文 | 说明 |
|------|------|------|
| Text-to-SQL | Text-to-SQL | 将自然语言文本转换为 SQL 查询语句的技术 |
| NL2SQL | Natural Language to SQL | 同 Text-to-SQL |
| Schema | Schema | 数据库的结构定义，包括表、字段、关系等 |
| BIRD | Benchmark for Instruction-based SQL | 大规模跨领域 Text-to-SQL 评测数据集 |
| EX | Execution accuracy | 执行准确率，衡量生成 SQL 执行结果与标准答案是否一致 |
| SFT | Supervised Fine-Tuning | 监督微调，一种模型训练方法 |
| vLLM | - | 高性能 LLM 推理引擎 |
| BM25 | Best Match 25 | 一种基于概率的检索排序算法 |
| MinHash | MinHash | 一种用于快速估计集合相似度的算法 |
| pass@k | - | 评测指标，k 个候选中至少有一个正确的比例 |
| Majority Voting | - | 多数投票，取多个采样结果中最常见的答案 |
