### 角色职责

| 角色 | 职责 | 在本阶段的定位 |
|------|------|----------------|
| **tester (Test Agent)** | 执行所有测试、编写Bug报告、验证修复、回归测试 | **主导角色**，控制测试流程和质量门禁 |
| backend-dev | 响应测试反馈、修复后端Bug、性能优化 | **响应角色**，按Test Agent指派的任务修复 |
| frontend-dev | 执行前端单元测试、修复前端Bug | **响应角色**，按Test Agent指派的任务修复 |

### 测试-反馈-修复循环机制

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  执行测试 │ → │ 发现问题 │ → │ 提交报告 │ → │ 分配修复 │ → │ 修复实现 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └─────┬────┘
     ↑                                                                │
     └──────────────────────── 回归测试通过 ←── 修复完成通知 ←──────────┘
```

**循环规则**:
1. **Test Agent 发现Bug** → 创建Bug报告 → 分配给对应开发人员
2. **开发人员修复** → 通知Test Agent → 提供修复说明
3. **Test Agent 验证** → 回归测试 → 通过则关闭，失败则重新打开
4. **循环直到** → 所有测试通过，无P0/P1 Bug

- Task 6.4 由 frontend-dev 执行，接受 tester 的测试反馈
- Task 6.5 由 backend-dev + frontend-dev 执行，**仅处理 tester 指派的Bug**

## 阶段6特有说明

### Test Agent 的主导地位

本阶段与其他阶段不同，**Test Agent (tester) 是主导角色**，而非辅助角色：

1. **质量门禁控制**
   - 只有 tester 可以判定测试是否通过
   - 只有 tester 可以关闭Bug报告
   - 只有 tester 可以批准进入下一阶段

2. **Bug生命周期管理**
   - Bug由 tester 创建，分配唯一ID
   - Bug状态流转：open → assigned → in_progress → fixed → verified → closed
   - 只有 tester 可以将状态改为 verified/closed

3. **开发人员定位**
   - backend-dev 和 frontend-dev 在本阶段是**响应角色**
   - 任务来源：tester 指派的Bug报告
   - 工作流程：接收Bug → 修复 → 通知 tester → 等待验证

### Bug报告和跟踪流程

**Bug报告创建**（由 tester 执行）:
```bash
# 在 docs/report/06-Phase6-Test/bugs/ 目录创建
# 命名规范: BUG-XXX-[简短描述].md
```

**Bug报告内容**:
```markdown
# Bug Report: BUG-001

| 字段 | 值 |
|------|-----|
| ID | BUG-001 |
| 标题 | 登录接口返回500错误 |
| 发现者 | tester |
| 优先级 | P0 |
| 模块 | backend/auth |
| 发现时间 | 2024-01-15 |
| 关联测试 | test_auth.py::test_login |

## 问题描述
当使用正确的用户名密码登录时，接口返回500错误。
```
