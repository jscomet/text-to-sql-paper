## 角色定义

### 1. e2e-lead (E2E 测试负责人)

**职责范围**:
- 制定 E2E 测试总体计划和策略
- 设计测试用例和测试数据
- 协调团队成员任务分配
- 审核测试报告和 Bug 报告
- 把控质量门禁，决定是否进入下一阶段

**输入**:
- 产品需求文档 (PRD)
- UI 设计文档
- API 接口文档
- 开发计划和进度

**输出**:
- E2E 测试计划文档
- 测试用例规范
- 汇总测试报告
- 质量评估结论

**关键决策点**:
- 测试范围确定
- 测试优先级排序
- Bug 严重级别认定
- 发布/阻断决策

---

### 2. test-lead (测试执行负责人)

**职责范围**:
- 执行具体的 E2E 测试用例
- 使用 MCP Playwright 工具进行测试
- 发现和记录 Bug
- 执行回归测试
- 编写详细的测试执行报告

**输入**:
- 测试计划文档
- 测试用例规范
- 前后端服务 (已部署)

**输出**:
- 测试执行记录
- 测试截图
- Bug 报告
- 回归测试报告

**使用工具**:
- `mcp__playwright__browser_navigate`
- `mcp__playwright__browser_type`
- `mcp__playwright__browser_click`
- `mcp__playwright__browser_snapshot`
- `mcp__playwright__browser_take_screenshot`
- `mcp__playwright__browser_wait_for`

**工作流程**:
```
1. 阅读测试用例规范
2. 准备测试环境
3. 执行测试步骤
4. 记录执行结果
5. 发现问题 → 截图 → 创建 Bug 报告
6. 完成测试 → 编写测试报告
7. 验证修复 → 回归测试
```

---

### 3. backend-dev (后端开发)

**职责范围**:
- 修复测试发现的 API 问题
- 排查后端逻辑错误
- 配合测试人员进行问题诊断
- 确保修复通过单元测试

**协作模式**:
- 接收 test-lead 指派的 Bug 报告
- 本地复现和修复
- 通知 test-lead 进行回归测试

---

### 4. frontend-dev (前端开发)

**职责范围**:
- 修复测试发现的 UI 问题
- 排查交互逻辑错误
- 配合测试人员进行问题诊断
- 确保修复通过单元测试

**协作模式**:
- 接收 test-lead 指派的 Bug 报告
- 本地复现和修复
- 通知 test-lead 进行回归测试

## 沟通协议

### 信息流转

```
测试计划 → e2e-lead → 全体成员
测试指令 → e2e-lead → test-lead
Bug 发现 → test-lead → backend-dev/frontend-dev
修复完成 → dev → test-lead
回归结果 → test-lead → e2e-lead
测试报告 → test-lead → e2e-lead
```

## 质量门禁

### P0 级 Bug (阻塞)

**定义**:
- 核心功能完全不可用
- 系统崩溃或数据丢失
- 安全漏洞
- 阻塞性错误

**处理**:
- 立即修复
- 修复前不能进入下一阶段
- 需要全量回归测试

### P1 级 Bug (严重)

**定义**:
- 主要功能有缺陷
- 有临时 workaround
- 用户体验严重受损

**处理**:
- 24 小时内修复
- 数量 < 5 个可进入下一阶段
- 需要相关模块回归测试

### P2 级 Bug (一般)

**定义**:
- 轻微功能问题
- 界面显示问题
- 性能优化点

**处理**:
- 可选修复
- 记录待后续版本处理

## 交付标准

### 测试计划完成标准

- [ ] 所有功能模块有测试用例
- [ ] 测试数据准备完成
- [ ] 测试环境配置文档化
- [ ] 团队成员理解测试范围

### 测试执行完成标准

- [ ] 所有 P0/P1 用例执行完毕
- [ ] 所有 Bug 已记录
- [ ] 截图和日志完整
- [ ] 测试报告已提交

### 阶段完成标准

- [ ] 无 P0 Bug
- [ ] P1 Bug < 5 个
- [ ] 所有修复已验证
- [ ] e2e-lead 批准
