# E2E 测试总计划

## 1. 测试概述

### 1.1 测试目标

对 Text-to-SQL 原型系统进行端到端测试，验证：
- 核心用户流程完整性
- 前后端集成正确性
- UI/UX 符合设计规范
- 边界情况和异常处理
- 性能基准达标

### 1.2 测试范围

| 范围项 | 包含 | 排除 |
|--------|------|------|
| 用户认证 | ✅ 登录/注册/登出 | ❌ 第三方 OAuth |
| 连接管理 | ✅ CRUD/测试连接 | ❌ SSL 证书验证 |
| 查询功能 | ✅ NL2SQL/SQL执行/结果展示 | ❌ 复杂图表 |
| 历史记录 | ✅ 查询历史/收藏/重新执行 | - |
| 评测功能 | ✅ 创建/执行/查看报告 | ❌ 分布式评测 |
| 个人设置 | ✅ 基本信息/API密钥 | ❌ 头像上传 |
| 响应式设计 | ✅ 桌面端/平板 | ❌ 手机端 (P2) |

## 6. 测试执行计划

### 6.1 冒烟测试清单

必须在 30 分钟内完成：

- [ ] 页面可正常访问
- [ ] 登录功能可用
- [ ] 创建连接成功
- [ ] 执行简单查询
- [ ] 查看历史记录

### 6.2 每日测试计划

**Day 1 - 冒烟测试 + Auth + Connection**
- 09:00-09:30 环境准备，冒烟测试
- 09:30-11:30 用户认证模块测试
- 11:30-12:00 上午总结，Bug 记录
- 14:00-17:00 连接管理模块测试
- 17:00-18:00 报告编写，Bug 分配

**Day 2 - Query + History**
- 09:00-12:00 查询功能模块测试
- 14:00-16:00 历史记录模块测试
- 16:00-18:00 Bug 修复，回归测试

**Day 3 - Evaluation + Settings + 边界测试**
- 09:00-12:00 评测功能测试
- 14:00-15:30 个人设置测试
- 15:30-17:30 边界情况测试
- 17:30-18:00 问题汇总

**Day 4 - 回归 + 性能**
- 09:00-12:00 Bug 修复回归
- 14:00-16:00 性能测试
- 16:00-18:00 最终报告

### 6.3 测试执行顺序

```
阶段 1: 冒烟测试
└── 核心流程验证

阶段 2: 模块测试 (按依赖顺序)
├── Auth 模块 (无依赖)
├── Connection 模块 (依赖 Auth)
├── Query 模块 (依赖 Auth + Connection)
├── History 模块 (依赖 Query)
├── Evaluation 模块 (依赖 Auth + Connection)
└── Settings 模块 (依赖 Auth)

阶段 3: 专项测试
├── 边界值测试
├── 异常情况测试
└── 并发测试 (如适用)

阶段 4: 回归测试
└── 全量 P0 用例

阶段 5: 性能测试
└── 响应时间和负载
```

## 8. 风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 测试环境不稳定 | 中 | 高 | 准备备用环境，Docker 化部署 |
| 发现大量 P0 Bug | 中 | 高 | 立即启动修复循环，延长测试周期 |
| 测试数据准备不足 | 低 | 中 | 提前准备，使用 fixtures |
| MCP 工具限制 | 低 | 中 | 准备本地 Playwright 备选 |
| 开发人员响应延迟 | 中 | 中 | 明确优先级，每日站会同步 |

## 12. 附录

### 12.1 用例 ID 编码规则

```
TC-{模块}-{序号}

模块代码:
- AUTH: 用户认证
- CONN: 连接管理
- QUERY: 查询功能
- HIST: 历史记录
- EVAL: 评测功能
- SET: 个人设置

示例: TC-AUTH-001 (用户认证模块第1个用例)
```

### 12.2 优先级定义

- **P0**: 核心功能，必须测试通过，阻塞发布
- **P1**: 重要功能，应测试通过，少量问题可接受
- **P2**: 一般功能，问题可延期处理
