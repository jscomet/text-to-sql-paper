# 高级推理功能 - 功能索引文档

## 1. 功能总览

高级推理功能模块为 Text-to-SQL 系统引入两种先进的**推理手段**，显著提升 SQL 生成的准确率和模型评估的完整性：

| 推理手段 | 核心思想 | 适用场景 |
|----------|----------|----------|
| **Sampling (多次采样)** | K 次独立采样，配合评测手段评估 | 评估模型潜力、探索性查询 |
| **Check-Correct (迭代修正)** | 生成-检查-修正迭代循环 | 生产环境高精度需求 |

### 1.1 为什么要增加这些功能

当前系统采用 **Single/Greedy**（单次生成）模式，每次生成只采样一次，存在以下局限：

1. **准确率天花板**：单次采样容易受随机性影响，无法发挥模型的最佳潜力
2. **缺乏自我修正**：生成错误后无法自动识别和修复
3. **评估不全面**：仅用 Greedy Search 评测会低估模型真实能力

**重要概念区分**：

| 类别 | 说明 | 具体手段 |
|------|------|----------|
| **推理手段** | 如何生成 SQL | Single/Greedy、Sampling、Check-Correct |
| **评测手段** | 如何评估结果 | Greedy Search、Pass@K、Majority Voting |
| **组合策略** | 推理+评测的完整方案 | vote@k、pass@k |

**组合策略说明**：

| 组合策略 | 推理手段 | 评测手段 | 说明 |
|----------|----------|----------|------|
| **vote@k** | Sampling (K次采样) | Majority Voting | K次采样后投票选择最佳SQL |
| **pass@k** | Sampling (K次采样) | Pass@K | K次采样中任一正确即算成功 |

参考 ICED-2026 论文代码：
- `infer.py` 控制**推理**（`--n` 参数控制采样次数，对应推理手段）
- `evaluate_bird.py` 控制**评测**（`--mode` 参数控制评测模式，对应评测手段）
- 组合策略示例：`--n 8` (Sampling) + `--mode major_voting` = **vote@8**

我们引入的高级功能：

- **Sampling 推理**：通过多次独立采样（temperature>0），配合 Pass@K 或 Majority Voting 评测，更准确地反映模型潜力
- **Check-Correct 推理**：借鉴 `lc_text_to_sql_pipeline.py` 的 Generate-Check-Correct 迭代流程，通过自我检查和修正提高最终输出质量

## 3. 快速开始

### 3.1 概念性介绍

#### 组合策略：pass@k (Sampling + Pass@K)

```
用户问题: "查询销售额最高的前10个客户"

Sampling (K=4) 推理 + Pass@K 评测流程:

推理阶段（Sampling）:
┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  采样 #1    │  │  采样 #2    │  │  采样 #3    │  │  采样 #4    │
│  SQL-A      │  │  SQL-B      │  │  SQL-C      │  │  SQL-D      │
│  temperature=0.8  │  temperature=0.8  │  temperature=0.8  │  temperature=0.8  │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │                │
       └────────────────┴────────────────┴────────────────┘
                          │
                          ▼
评测阶段（Pass@K）:
┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  执行 SQL-A │  │  执行 SQL-B │  │  执行 SQL-C │  │  执行 SQL-D │
│  (错误)     │  │  (正确)     │  │  (错误)     │  │  (错误)     │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │                │
       └────────────────┴────────────────┴────────────────┘
                          │
                          ▼
                    ┌─────────────┐
                    │ Pass@K=通过 │  (K次中任一正确即算成功)
                    └─────────────┘
```

**组合策略配置**（pass@k）：
- `n` (推理): 采样次数，支持 1, 4, 8, 16（默认 8）
- `temperature` (推理): 采样温度，建议 0.7-0.9（默认 0.8）
- `eval_mode` (评测): 设为 "pass_at_k"
- `k` (评测): 与采样次数一致

> **注意**: pass@k 是组合策略，需要同时配置推理参数（n, temperature）和评测参数（eval_mode, k）

#### 组合策略：vote@k (Sampling + Majority Voting)

```
用户问题: "查询销售额最高的前10个客户"

vote@k (K=4) 组合策略流程:

推理阶段（Sampling）:
┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  采样 #1    │  │  采样 #2    │  │  采样 #3    │  │  采样 #4    │
│  SQL-A      │  │  SQL-B      │  │  SQL-C      │  │  SQL-D      │
│  结果=X     │  │  结果=X     │  │  结果=Y     │  │  结果=X     │
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │                │
       └────────────────┴────────────────┴────────────────┘
                          │
                          ▼
评测阶段（Majority Voting）:
┌─────────────────────────────────────────────────────────────┐
│  结果统计:                                                  │
│    结果 X: 3 票 (SQL-A, SQL-B, SQL-D)                      │
│    结果 Y: 1 票 (SQL-C)                                    │
│                                                             │
│  胜出者: 结果 X (票数最多)                                  │
│  返回 SQL: SQL-A (该组第一个)                               │
└─────────────────────────────────────────────────────────────┘
```

**组合策略配置**（vote@k）：
- `n` (推理): 采样次数，支持 1, 4, 8, 16（默认 8）
- `temperature` (推理): 采样温度，建议 0.7-0.9（默认 0.8）
- `eval_mode` (评测): 设为 "majority_vote"
- `k` (评测): 与采样次数一致

> **注意**: vote@k 是 ICED-2026 中常用的组合策略，`--n 8` + `--mode major_voting` = vote@8

#### Check-Correct 推理

```
用户问题: "计算每个部门的平均工资"

Check-Correct 迭代流程 (最多3轮):

第1轮:
┌──────────┐      ┌──────────┐      ┌──────────┐
│ Generator│─────▶│ Checker  │─────▶│ 结果=NO  │
│ 生成SQL1 │      │ 检查错误 │      │ 语法错误 │
└──────────┘      └──────────┘      └────┬─────┘
                                         │
第2轮:                                   ▼
┌──────────┐      ┌──────────┐      ┌──────────┐
│Corrector │─────▶│ Checker  │─────▶│ 结果=NO  │
│ 修正SQL2 │      │ 检查错误 │      │ 语义错误 │
└──────────┘      └──────────┘      └────┬─────┘
                                         │
第3轮:                                   ▼
┌──────────┐      ┌──────────┐      ┌──────────┐
│Corrector │─────▶│ Checker  │─────▶│ 结果=YES │
│ 修正SQL3 │      │ 检查错误 │      │ 通过     │
└──────────┘      └──────────┘      └────┬─────┘
                                         │
                                         ▼
                                   ┌──────────┐
                                   │ 返回SQL3 │
                                   └──────────┘
```

**推理配置参数**（Check-Correct）：
- `max_iterations`: 最大迭代次数，支持 1-5（默认 3）
- `error_types`: 错误分类检查（syntax, execution, semantic）
- `temperature`: 修正时的采样温度（默认 0.7）

**评测配置参数**：
- `eval_mode`: 设为 "check_correct"
- 评测最终迭代结果的准确性

### 3.2 使用场景建议

| 场景 | 推荐组合策略 | 推理手段 | 评测手段 | 说明 |
|------|-------------|----------|----------|------|
| 日常查询 | Single/Greedy | Single/Greedy | Greedy Search | 响应快，资源消耗低 |
| 重要报表 | Check-Correct | Check-Correct | 最终迭代结果评测 | 确保准确性，容忍较长耗时 |
| 模型评测 | **pass@k** | Sampling | Pass@K | 评估模型潜力上限 |
| 提高准确率 | **vote@k** | Sampling | Majority Voting | 基于执行结果的投票选择 |
| 生产部署 | Check-Correct | Check-Correct | 最终迭代结果评测 | 自动修正减少人工介入 |

## 4. 与现有功能对比

### 4.1 推理手段对比

| 特性 | Single/Greedy | Sampling | Check-Correct |
|------|---------------|----------|---------------|
| **temperature** | = 0 | > 0 (通常0.7-0.9) | 可变（逐轮调整） |
| **生成次数** | 1 次 | K 次 | 最多迭代数次 |
| **并行执行** | 否 | 是（K个并发） | 否（串行迭代） |
| **输出数量** | 1 个 SQL | K 个候选 SQL | 1 个最终 SQL |
| **平均耗时** | ~3s | ~3s（并行） | ~3s × 迭代数 |
| **资源消耗** | 低 | 高（K倍） | 中（2-3倍） |
| **适用场景** | 日常查询 | 评测/研究 | 生产环境 |

### 4.2 评测手段对比

| 评测手段 | 配合推理手段 | 评测逻辑 | 适用场景 |
|----------|-------------|----------|----------|
| **Greedy Search** | Single/Greedy | 单次结果直接评测 | 日常查询准确性 |
| **Pass@K** | Sampling | K个候选中任一正确即算通过 | 评估模型潜力上限 |
| **Majority Voting** | Sampling | 基于执行结果投票选择最佳 | 提高实际应用准确率 |
| **Check-Correct 评测** | Check-Correct | 评测最终迭代结果 | 评估迭代修正效果 |

### 4.3 组合策略对比

| 组合策略 | 推理手段 | 评测手段 | 输出 | 适用场景 |
|----------|----------|----------|------|----------|
| **Single/Greedy** | Single/Greedy | Greedy Search | 1个SQL | 日常查询 |
| **vote@k** | Sampling (K次) | Majority Voting | 1个SQL（投票胜出） | 提高准确率 |
| **pass@k** | Sampling (K次) | Pass@K | 通过/失败 | 模型评测 |
| **Check-Correct** | Check-Correct | 最终迭代评测 | 1个SQL | 高精度需求 |

### 4.4 组合策略效果预期

基于 ICED-2026 实验数据（BIRD-dev 数据集）：

| 组合策略 | 配置 | 预期 EX 准确率 | 相对提升 | 平均耗时 |
|----------|------|----------------|----------|----------|
| Single/Greedy | temperature=0 | ~60% | 基准 | ~3s |
| **vote@8** | n=8, temperature=0.8 | ~70% | +10% | ~10s |
| **pass@8** | n=8, temperature=0.8 | ~72% | +12% | ~10s |
| Check-Correct | max_iter=3 | ~68% | +8% | ~20s |

**说明**：
- vote@k 和 pass@k 使用相同的推理配置（Sampling），但评测方式不同
- vote@k 返回投票胜出的 SQL，用于实际应用
- pass@k 返回是否任一通过，用于模型能力评估
